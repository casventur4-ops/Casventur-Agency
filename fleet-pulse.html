FleetPulse Lite

FleetPulse Lite es una solución de telemática ligera para flotas pequeñas y medianas. Convierte el smartphone del conductor en una fuente de señales de viaje para generar historial, eventos de manejo, score por viaje y alertas básicas para el gerente de flota.

Status: working baseline / build-ready Android-first para v1 operativa App móvil + dashboard web Paddle billing Privacy & governance posture activa

Usuario primario

Gerente o dueño de flota SMB, supervisor de conductores y operación con 5–50 vehículos.

Usuario secundario

Conductor o repartidor que usa la app móvil durante un viaje activo.

Tesis del producto

Para muchas flotas SMB, algo útil, rápido y sin hardware vale más que una plataforma pesada con instalación física.

Problema que resuelve

Muchas flotas pequeñas hoy no usan telemática, o dependen de llamadas, intuición y WhatsApp, o consideran demasiado costosas las soluciones con hardware. FleetPulse reduce esa fricción con una propuesta BYOD, rápida y más ligera.

Lo que el producto sí es

Telemática ligera, scoring básico de conducción, supervisión de viajes, disciplina operacional, coaching y visibilidad básica para la flota.

Lo que no promete en v1

No se presenta como caja negra forense, herramienta actuarial, TMS, dashcam, vigilancia permanente fuera de viaje ni sistema de sanción automática.

Alcance v1

Qué entra y qué se deja fuera para proteger claridad

In scope

  • App móvil para conductor
  • Autenticación
  • Trip start / stop
  • Captura de GPS y sensores durante viaje activo
  • Detección de eventos básicos
  • Score por viaje
  • Dashboard web para gerente
  • Historial de viajes, mapa, alertas y export básico
  • Multitenancy por flota
  • Suscripción y billing

Out of scope

  • Seguros y pricing actuarial
  • Hardware dedicado
  • Dashcam o video
  • Routing y dispatch
  • Geofencing avanzado
  • API pública e integraciones ERP
  • Gamificación
  • Coaching con IA generativa
  • Scoring laboral o disciplinario automatizado

Decisiones cerradas

  • Android-first: la operación seria de v1 se valida primero en Android.
  • Viaje explícito: start/stop manual en v1, no detección totalmente automática.
  • Speeding advisory-first: no se vende como verdad perfecta sin cobertura validada.
  • Score honesto: señal de comportamiento y coaching, no verdad jurídica.
Arquitectura resumida

Cuatro capas para una operación ligera pero seria

1. Mobile app

Captura viaje activo, muestra estado del trayecto y score preliminar para el conductor.

2. Backend transaccional

Recibe trip summaries, recalcula el score autoritativo, persiste eventos y decide alertas.

3. Dashboard web

Muestra conductores, viajes, score, mapas, tendencias y exportes para la flota.

4. Billing + servicios externos

Paddle para suscripción, proveedor de mapas y, cuando aplique, proveedor de speed limits.

Stack sugerido

React Native con Expo, Next.js, Supabase, Paddle y componentes server-side según el caso.

Regla clave

El preview score móvil es informativo; el score final del backend es la versión autoritativa.

Score engine

Un motor simple, explicable y versionable

El score v1 resume riesgo relativo por viaje para coaching y priorización. Los eventos base son hard brake, hard accel, sharp turn y speeding con enfoque advisory-first.

Principio técnico

Ningún evento debe depender ciegamente de una sola señal ruidosa. La lógica cruza acelerómetro, giroscopio, velocidad GPS y contexto temporal.

Bandas del score

A: 90–100 · B: 75–89 · C: 60–74 · D: 45–59 · F: menos de 45. La salida debe explicar qué pasó, cuántas veces, dónde y cuánto afectó el score.

Gobernanza del score

Cada viaje debe guardar versión del score, resumen de entradas y deducciones. Si el score no es explicable, no es confiable.

Privacidad & seguridad

El producto se diseña con una postura de governance real

Minimización

Se recolecta la mínima cantidad de datos necesaria para entregar el servicio. No se plantea audio, video, contactos, mensajes ni tracking permanente fuera de viaje.

Retención y transparencia

Eventos y viajes con retención por defecto de 12 meses, logs operativos 30–90 días, y explicaciones claras al conductor sobre cuándo se recolecta ubicación y qué no se monitorea.

Postura legal y de seguridad

No se asume automáticamente un rol legal absoluto. Se prevén DPA, matriz de retención, inventario de subprocesadores, separación por flota, TLS, cifrado en reposo y authz por objeto.

FleetPulse se cuenta con una regla de honestidad central: no prometer equivalencia iOS/Android sin prueba física, no inflar precisión del score y no vender speeding como verdad perfecta mientras siga en postura advisory-first.
Ejecución

Plan de build por slices

La agencia deja el proyecto listo para construirse con control. El plan no persigue features por glamour; persigue reducción de incertidumbre técnica, comercial y de privacidad.

Slice 0
Base del sistema
Repo, auth, multitenancy, skeleton móvil y web, estructura inicial y baseline documental cerrada.
Slice 1
Trip recording
Start/stop, permisos, captura GPS, timer, estados y cola offline.
Slice 2
Eventos v1
Hard brake, hard accel y sharp turn con detección local básica.
Slice 3
Score preview
Detector local, deducciones y explicabilidad básica.
Slice 4
Backend score engine
Recalcular score, persistir viajes y eventos, versionado y respuesta consistente.
Slice 5
Dashboard admin
Conductores, viajes, detalle de ruta, mapas y filtros básicos.
Slice 6
Billing
Paddle checkout, webhook, estados de suscripción y gating.
Slice 7
Hardening
Privacidad, seguridad, release gates, auditability y soporte inicial.

Roadmap
